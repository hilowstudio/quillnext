# 02 — Data Model (Prisma schema + migrations)

> Code-truth reference. Verified against source on 2026-06-15. Where this doc
> and any prose/markdown in the repo disagree, **the schema and migration SQL win**.
> Everything below is cited to `prisma/schema.prisma:<line>` or the migration SQL.

## Purpose & role in the app

This subsystem is the **single source of truth for the relational data layer** of
QuillNext / "Quill & Compass" — a Next.js 16 / React 19 / Prisma 7 faith-based
homeschool & family-discipleship platform on Postgres (Supabase) with pgvector.

It consists of exactly three owned artifacts:

- `prisma/schema.prisma` (1387 lines) — 56 models + 18 enums defining curriculum,
  classrooms, students, AI-generated resources, assessments, scheduling, and a
  large "Family Discipleship" suite.
- `prisma/migrations/00000000000000_extensions_rls/migration.sql` — a hand-authored
  **baseline** migration that installs the `vector` extension and an event trigger
  that auto-enables Row Level Security on every new `public` table.
- `prisma/migrations/00000000000001_init/migration.sql` (1517 lines) — the generated
  `init` migration creating all enums, tables, indexes, and foreign keys.
- `prisma/migrations/migration_lock.toml` — pins provider `postgresql`.

The generated client is emitted to `src/generated/client` (`schema.prisma:4`) and
instantiated once in `src/server/db.ts` as the `db` singleton. Every server action,
route handler, and query in other subsystems imports `db` from there.

The defining architectural fact of this layer: **multi-tenancy is expressed in the
Prisma models as `organizationId` but stored in Postgres as the column `account_id`
on almost every tenant-owned table** (via `@map("account_id")`). Two models break
this convention and use `organization_id` instead (see "Data models & tenancy").

## File-by-file reference

### `prisma/schema.prisma`

- **Role:** Declarative datamodel. Defines 56 models, 18 enums, all relations,
  indexes, `@map`/`@@map` name overrides, and `onDelete` behavior.
- **Generator / datasource:**
  - `generator client` uses the new `prisma-client` provider (Prisma 7), output
    `../src/generated/client`, `binaryTargets = ["native"]` (`schema.prisma:1-6`).
  - `previewFeatures = ["postgresqlExtensions"]` is **commented out** (`schema.prisma:3`)
    — so extension management is NOT done via Prisma; it is done manually in the
    baseline migration instead.
  - `datasource db` is `postgresql` with **no inline `url`/`directUrl`** (`schema.prisma:8-10`);
    the connection string is injected via the pg adapter in `src/server/db.ts`
    (`DATABASE_URL`). Note: a `directUrl` is therefore NOT declared in the schema even
    though `.env` defines `DIRECT_DATABASE_URL`.
- **Server/client:** N/A (schema file; compiled to a server-only client).
- **Key structural notes (cite):**
  - Vector columns: `Book.embedding Unsupported("vector")?` (`schema.prisma:670`) and
    `VideoResource.embedding Unsupported("vector")?` (`schema.prisma:720`). Prisma
    cannot type these; they are read/written via raw SQL (see end-to-end flows).
  - `StudentCatechismProgress.catechismId` (`schema.prisma:1376`) maps to `catechism_id`
    but has **NO relation** to the `Catechism` model — it stores the catechism *slug*
    (`Catechism.code`, e.g. `"wsc"`), per the comment at `schema.prisma:34-36`, not the
    `Catechism.id`. This is intentional but is a latent foot-gun (no referential integrity).
  - `CourseBlock` is self-referential via `parentBlock`/`childBlocks`
    (`"BlockHierarchy"`, `schema.prisma:539-540`) and carries six optional content FKs
    (`book`, `video`, `article`, `document`, `resource`, plus topic/subtopic) making it
    the polymorphic content hub.
  - `CourseBlock.bookChapterId` (`schema.prisma:531`) and `CourseBlock.sourceBundleId`
    (`schema.prisma:522`) are plain `String?` columns with **no FK / no relation** —
    used as soft references (set in `src/app/actions/explode-bundle.ts` and
    `src/app/api/courses/[id]/blocks/route.ts`).

### `prisma/migrations/00000000000000_extensions_rls/migration.sql`

- **Role:** Hand-written **baseline** that must run *before* `init`. Not generated by
  Prisma from the datamodel — it reproduces out-of-band DB objects so a
  `prisma migrate deploy` can build the canonical DB from scratch.
- **What it does:**
  1. `CREATE EXTENSION IF NOT EXISTS "vector";` (line 7) — required by the
     `books.embedding` / `video_resources.embedding` columns the next migration creates.
  2. Defines `public.rls_auto_enable()` — a `SECURITY DEFINER` event-trigger function
     that, on every `CREATE TABLE` in `public`, runs `ALTER TABLE ... ENABLE ROW LEVEL
     SECURITY` (lines 12-36).
  3. Creates event trigger `ensure_rls ON ddl_command_end` **guarded** in a `DO` block
     so a restricted role that can't create event triggers degrades to a `NOTICE`
     instead of failing the migration (lines 40-50).
- **Critical implication:** RLS gets *enabled* on every table, but this migration
  creates **zero `CREATE POLICY` statements**, and a grep of the entire repo finds
  no `CREATE POLICY` anywhere. See "Auth / security posture".

### `prisma/migrations/00000000000001_init/migration.sql` (1517 lines)

- **Role:** The generated initial migration. Creates `public` schema, 18 enum types
  (lines 4-65), all 56 tables (lines 67-978), all indexes (lines 980-1185), and all
  foreign keys (lines 1187-1516).
- **Vector columns appear as bare `vector`** (no dimension): `books.embedding vector`
  (line 576), `video_resources.embedding vector` (line 617). Because the type is
  unconstrained, the column accepts any-dimension vectors; the app code assumes 1536
  dims (see drift note).
- **`student_catechism_progress`** is created (lines 967-978) with a unique index on
  `(student_id, catechism_id)` (line 1185) and a FK **only** on `student_id`
  (line 1515) — confirming there is no `catechism_id` FK at the DB level.
- This migration is the authoritative record of `onDelete` behavior; the schema's
  Prisma-level `onDelete` annotations are translated into the `ON DELETE ...` clauses
  here. They match (no schema-vs-migration drift in cascade behavior was found).

### `prisma/migrations/migration_lock.toml`

- Pins `provider = "postgresql"`. Trivial; do not edit by hand.

## Data models & tenancy

### Tenancy mapping (the most important fact)

The tenant root is **`Organization`** (`organizations` table, `schema.prisma:102`).
`OrganizationType` enum: `PARENT_INSTRUCTOR | MICROSCHOOL_COOP | CHURCH_PRIVATE_SCHOOL`
(`schema.prisma:912`).

Tenant-owned models carry the tenant FK as Prisma field **`organizationId`** but the
DB column name differs:

| DB column | Models (Prisma `organizationId @map(...)`) |
|---|---|
| **`account_id`** | `User` (`:147`), `Classroom` (`:215`), `Student` (`:281`), `Course` (`:480`), `Book` (`:652`), `VideoResource` (`:702`), `Resource` (`:733`), `Article` (`:1091`), `DocumentResource` (`:1116`), `StudentScheduleItem` (`:1298`), `CustomEvent` (`:1334`) |
| **`organization_id`** | `Transcript` (`:127`), `CurriculumSpec` (`:769`) — **the two exceptions** |

`User.organizationId` is **nullable** (`account_id TEXT` nullable, `schema.prisma:147`,
migration line 166) — a user can exist before being attached to an org. Everything
else that has a tenant FK is non-null.

Tenancy is **not** enforced by the database. There is no Postgres policy and the app
connects as a BYPASSRLS role (see security posture). Isolation is enforced **only** by
application code passing `where: { organizationId }` (pattern visible throughout
`src/server/queries/students.ts`, e.g. `getStudentById` re-checks
`student.organizationId !== organizationId` at `students.ts:186`). The org id comes from
`getCurrentUserOrg()` in `src/lib/auth-helpers.ts`, which reads the NextAuth session and
loads `user.organizationId`.

### Domain groupings (ER-style overview)

**1. Identity & Auth (NextAuth/Auth.js adapter tables)**
`User` (`:140`), `Account` (`:175`), `Session` (`:194`), `VerificationToken` (`:204`).
`User.role` is `UserRole` (`OWNER|TEACHER|ADMIN|PARENT`, default `PARENT`, `:146`).
`Account`/`Session` cascade-delete with `User`.

**2. Organization & Classroom**
`Organization` → `Classroom` (`:213`), `ClassroomInstructor` (`:245`, join User↔Classroom
with `instructorPin` + `InstructorRole`), `ClassroomHoliday` (`:264`),
`ClassroomStudent` (`:336`, composite-PK join). Classroom carries pedagogy config:
`EducationalPhilosophy` (17 values, `:925`), `FaithBackground` (19 values, `:945`),
school-year dates, `school_days_of_week` JSON, daily times.

**3. Students & Learner Profile**
`Student` (`:279`) is the second hub (after Organization). Has special-needs columns
(`support_intensity`, `support_labels String[]`, `support_profile Json`,
`learning_difficulties`, `:290-293`), `avatarConfig Json`. One-to-one
`LearnerProfile` (`:350`, unique `student_id`) stores AI personality/learning-style/
interests JSON. `SafetyFlag` (`:317`) is an AI-moderation table (severity/category/
reasoning/`implicated_caregiver`) cascade-tied to Student.

**4. Curriculum Taxonomy (reference data, mostly org-agnostic)**
`Subject` (`:366`) → `Strand` (`:386`) → `Topic` (`:410`) → `Subtopic` (`:429`) →
`Objective` (`:448`). `GradeBand` (`:466`). These use `code`/`uuid` natural keys and
`onDelete: Restrict` (cannot delete a parent with children). They are NOT tenant-scoped.

**5. Courses & Blocks**
`Course` (`:478`, tenant-scoped) → `CourseBlock` (`:518`, self-nesting, polymorphic
content hub) → `Activity` (`:557`) → `ActivityObjective` (`:578`, join to Objective).
`CourseStudent` (`:503`, enrollment join, `CourseStudentStatus`), `CourseProgress`
(`:895`, per-student progress, unique `(course,student)`). Block taxonomy enums:
`CourseBlockKind` (`:987`), `ActivityType` (`:995`).

**6. Assessments & Grading**
`Assessment` (`:590`) → `AssessmentItem` (`:613`) → `AssessmentItemResponse` (`:875`).
`AssessmentAttempt` (`:849`, student attempt, score, `letterGrade`, `gradingMethod`).
Enums: `AssessmentScopeKind`, `AssessmentType`, `AssessmentItemType`, `AttemptStatus`,
`GradingMethod` (`:1005-1079`). `total_points`/`points`/scores are `Decimal(65,30)`.

**7. Content Resources & AI Generation**
`Book` (`:650`, external books, **`embedding vector`**), `VideoResource` (`:700`,
YouTube, **`embedding vector`**, unique `youtube_video_id`), `Article` (`:1089`),
`DocumentResource` (`:1114`, uploaded files). `ResourceKind` (`:630`, catalog of
generator types). `Resource` (`:731`) is the generated-output table with five
`generatedFrom*` source FKs (book/video/article/document) + `generatedForStudentId`.
`BookGeneratedMaterial` (`:684`) links a Book + ResourceKind → generated Resource.
Extraction lifecycle via `ExtractionStatus` enum (`:1045`).

**8. AI Curriculum Compiler ("bundles")**
`CurriculumSpec` (`:767`, uses `organization_id`) → `CurriculumBundle` (`:783`) →
`Resource` (via `curriculumBundleId`, `:761`). `CurriculumBundle` has **untyped
string** `status` ("COMPILING"/"COMPLETED"/"FAILED") and `parentBundleId` lineage
(plain string, no FK). Driven by `src/inngest/functions/compile-curriculum.ts` and
`src/app/actions/explode-bundle.ts`.

**9. Scheduling / Planner**
`StudentScheduleItem` (`:1296`, polymorphic-ish: optional `courseBlockId`/`activityId`,
`ScheduleItemStatus` enum `:1365`) and `CustomEvent` (`:1332`, with iCal `recurrence_rule`
+ `parentEventId` soft self-ref). Both tenant-scoped via `account_id`.

**10. Assignments**
`ResourceAssignment` (`:799`) — assigns a Resource to any of course/block/activity/
assessment/student. Heavily indexed (`:822-826`).

**11. Family Discipleship Suite** (`schema.prisma:1136` onward)
Mostly **user-scoped (not org-scoped)** — FKs point at `User`, not `Organization`:
`Devotional` (`:1138`, global content seeded from `prisma/data/devotionals.json`),
`GratitudeJournal` (`:1151`), `DevotionalReflection` (`:1165`), `LocalChurchNotes`
(`:1184`), `PrayerJournalEntry` (`:1208`, user **and** optional student),
`PrayerCategory` (`:1235`, global), `BibleMemory` (`:1249`, user **or** student, 8-step
mastery tracking), `BibleMemoryFolder` (`:1280`), `StudentCatechismProgress` (`:1373`,
student-scoped, slug-keyed). Reference content: `Catechism`/`CatechismQuestion`
(`:37`/`:55`), `CommentaryChapter`/`CommentarySection` (`:72`/`:87`, Matthew Henry),
`County` (`:17`, "Neighbor Love" feature, full record in `data Json`).

### Cascade blast radius (verified from FK SQL, lines 1187-1516)

- **Deleting an `Organization`** cascades to: users, classrooms, students, courses,
  books, video_resources, resources, articles, document_resources,
  student_schedule_items, custom_events, transcripts. Then each of those cascades
  further (e.g. deleting students cascades to learner_profiles, course_students,
  classroom_students, safety_flags, activity_progress, assessment_attempts,
  resource_assignments, prayer/bible-memory/catechism rows, transcripts). **An org
  delete is effectively a full-tenant wipe.**
- **Deleting a `User`** cascades to: accounts, sessions, AND **the entire org** —
  because `users.account_id` FK is `ON DELETE CASCADE` *and* org→user is also CASCADE,
  but the dangerous direction is org-delete → users. Also cascades the user's
  discipleship journals (gratitude, reflections, church notes, prayer entries,
  bible memory). Note: `User` is also referenced by many "creator" relations
  (`createdByUserId`, `addedByUserId`, `assignedByUserId`) which are **`ON DELETE
  RESTRICT`** (e.g. `courses_created_by_user_id_fkey`, line 1254) — so you generally
  **cannot** delete a user who created courses/books/resources/assessments without
  first reassigning or deleting those. This RESTRICT/CASCADE asymmetry is a real
  operational hazard.
- **Deleting a `Student`** cascades broadly (progress, attempts, enrollments,
  safety_flags, schedule items, transcripts, prayer/bible-memory). Generated
  artifacts soften it: `book_generated_materials.generated_for_student_id` and
  `resources.generated_for_student_id` are `ON DELETE SET NULL` (lines 1344, 1371).
- **Content FKs on `course_blocks`** (book/video/article/document/resource/topic/
  subtopic/parent) are all `ON DELETE SET NULL` (lines 1272-1296) — deleting a book
  nulls the block's `book_id` rather than deleting the block.
- **Taxonomy** (`subject`/`strand`/`topic`/`subtopic`/`objective`) parents are
  `ON DELETE RESTRICT`; `resource_kinds`/`courses`/`books`/etc. referencing
  subject/strand are `ON DELETE SET NULL` or `RESTRICT` (subject is RESTRICT, strand
  is SET NULL — e.g. `books_subject_id_fkey` RESTRICT line 1338, `books_strand_id_fkey`
  SET NULL line 1335).

### Vector / embedding columns

- `books.embedding` and `video_resources.embedding`, both `Unsupported("vector")?`
  in schema (`:670`,`:720`), bare `vector` in SQL (migration lines 576, 617).
- Read/written exclusively via raw SQL in `src/lib/utils/vector.ts`
  (`generateBookEmbedding`, `searchBooks`, `findSimilarBooks`, `searchVideos`,
  `generateVideoEmbedding`) using cosine distance operator `<=>` and casting a
  JS array to `::vector`.
- Embedding model: `google.textEmbeddingModel("gemini-embedding-2")` in
  `src/lib/ai/config.ts` (set 2026-06-16; was `text-embedding-004`). Stored at
  **1536 dims** via `outputDimensionality` (`embeddingProviderOptions`), with
  asymmetric `taskType` (RETRIEVAL_DOCUMENT for stored content, RETRIEVAL_QUERY for
  searches). The `vector` column is still dimensionless (no DB-level check), so the
  1536 is enforced only in app code (`EMBEDDING_DIMENSIONS`). The old "1536 dims" drift
  is now resolved — it's genuinely 1536. Both vector tables were empty at swap time, so
  no re-embed was needed.

## Entry points & end-to-end flows

This subsystem has no routes of its own; it is consumed by every other subsystem via
the `db` singleton (`src/server/db.ts`). Representative flows that exercise the data
model:

**A. Book ingestion + embedding (semantic search)**
`POST /api/library/books` (`src/app/api/library/books/route.ts`) → resolves org via
`getCurrentUserOrg` → `db.book.create({ account_id })` → best-effort
`generateBookEmbedding(book.id, text)` (`vector.ts`) issues
`UPDATE "books" SET embedding = ...::vector`. A failure is swallowed and logged
(`route.ts:95-108`) — the book stays searchable in the list UI but **invisible to
semantic search until re-embedded**. Search side: `searchBooks(query)` → embed query →
`SELECT ... 1 - (embedding <=> $vec::vector) AS similarity ... WHERE similarity > 0.5`.

**B. Tenant-scoped read**
Any server query (e.g. `getStudentById(studentId, organizationId)` in
`src/server/queries/students.ts:179`) filters by `organizationId` and additionally
re-validates ownership in JS. This manual filtering **is the only tenant boundary**.

**C. AI curriculum compile**
`CurriculumSpec` created → Inngest `compile-curriculum` builds a `CurriculumBundle`
(status string transitions COMPILING→COMPLETED/FAILED) → emits `Resource` rows linked
via `curriculumBundleId` → `explode-bundle.ts` materializes `CourseBlock`s tagged with
`sourceBundleId`.

**D. Migration / DB build**
`prisma migrate deploy` runs `00000000000000_extensions_rls` (vector ext + RLS trigger)
then `00000000000001_init` (all tables, which the trigger auto-RLS-enables).
Seeding is via `prisma/seed*.ts` (`db:seed*` npm scripts; `prisma.config.ts` sets
`seed: 'tsx prisma/seed.ts'`).

## Auth / security posture

- **No database-enforced multi-tenancy.** RLS is *enabled* on every `public` table
  by the `rls_auto_enable` event trigger, but the repo contains **zero `CREATE POLICY`
  statements** (verified by grep across `src` and `prisma`). RLS-enabled + no-policies
  normally means *deny all* — except the app connects as the Supabase **`postgres`**
  superuser role (`DATABASE_URL` user `postgres.<ref>`), which has **`BYPASSRLS`**.
  Net effect: **RLS is a no-op for the application; tenant isolation depends entirely
  on application-level `where: { organizationId }` filtering.** Any query that forgets
  the org filter leaks across tenants. This is the single biggest security note for
  this subsystem.
- **Org resolution** is centralized in `getCurrentUserOrg()`
  (`src/lib/auth-helpers.ts`) — reads NextAuth session, looks up `user.organizationId`,
  throws if unauthenticated. It returns `organizationId` which may be `null` for a
  not-yet-onboarded user (schema allows nullable `User.account_id`); callers must
  handle the null.
- **`SafetyFlag`** is security/child-safety sensitive (AI moderation of student chat,
  including `implicated_caregiver`). It is Student-scoped but **not** directly
  Organization-scoped — access control must be derived through the Student's org.
- **OAuth secrets** (`refresh_token`, `access_token`, `id_token`) are stored
  unencrypted in the `accounts` table (`schema.prisma:181-186`) — standard for the
  Auth.js Prisma adapter, but worth noting they are plaintext at rest.
- The migration's RLS function is `SECURITY DEFINER` with `search_path = pg_catalog`
  (good hardening) and the event-trigger creation is guarded for least-privilege roles.

## Risks, drift, dead-code & half-built

1. **RLS theater (high).** Every table has RLS enabled but no policies exist, and the
   app role is BYPASSRLS — so the RLS infrastructure provides *no* actual protection.
   A reader could be falsely reassured by "RLS enabled". Isolation is 100% app-level.
2. **Embedding dimension (resolved 2026-06-16; was a drift note).** Now `gemini-embedding-2`
   stored at **1536 dims** (pinned in app code via `EMBEDDING_DIMENSIONS` + `outputDimensionality`).
   The `vector` column remains dimensionless (no DB constraint), so consistency is enforced only
   in code — changing `EMBEDDING_DIMENSIONS` requires re-embedding all rows, and mixing dimensions
   in a column would make cosine queries error. (1536 stays ≤2000, so an HNSW/IVFFlat index can be
   added later if desired.)
3. **`StudentCatechismProgress` has no FK to `Catechism` (medium).** `catechism_id`
   actually stores the catechism *slug* (`Catechism.code`), not `Catechism.id`
   (`schema.prisma:34-36`, FK absent at DB level — migration line 1515 only adds the
   student FK). No referential integrity; renaming/removing a catechism code silently
   orphans progress rows.
4. **`CurriculumBundle.status`/`parentBundleId`/`CurriculumSpec` are loosely typed
   (medium).** `status` is a free `String` (COMPILING/COMPLETED/FAILED by convention,
   `schema.prisma:791`) instead of an enum; `parentBundleId` is a plain string with no
   self-FK. Easy to write an invalid status; lineage isn't integrity-checked.
5. **Best-effort embedding write swallows failures (medium).** A book/video that fails
   embedding is silently absent from semantic search with only a console log
   (`books/route.ts:95-108`). There is no `embedding_status` column to detect or
   re-drive these, so they are invisible to ops.
6. **`User` delete vs creator RESTRICT asymmetry (medium, operational).** Many tables
   reference `User` via RESTRICT creator FKs (`created_by_user_id`, `added_by_user_id`,
   `assigned_by_user_id`). You cannot delete a user who authored content, yet
   org-deletion CASCADEs users — making org teardown order-sensitive and error-prone.
7. **Two tenancy column conventions (low-medium).** Most tenant tables use
   `account_id`; `Transcript` and `CurriculumSpec` use `organization_id`. Raw SQL
   (vector queries, any hand-written SQL) must remember which is which; mixing them is
   a silent bug source.
8. **`Decimal(65,30)` everywhere for points/scores (low).** Prisma's default Decimal
   precision (65,30) is used for `total_points`, `points`, scores, `points_earned`
   (migration lines 513, 530, 719-720, 737-738). Wildly over-precise for grades;
   harmless but odd, and serializes to string in JS.
9. **`Objective.gradeLevel` is camelCase in DB (low/drift).** Most columns are
   snake_cased, but `objectives.gradeLevel` and several enum FK columns
   (`classrooms.educationalPhilosophy`, `classrooms.faithBackground`,
   `activities.activityType`) are left camelCase in the actual table (migration lines
   406, 216, 218, 486) because those fields lack `@map`. Inconsistent naming; matters
   for raw SQL.
10. **Schema datasource omits `url`/`directUrl` (low).** Connection comes only from
    the pg adapter env. `prisma migrate`/`studio` need the URL passed via env, and a
    pooled (pgbouncer) `DATABASE_URL` is unsuitable for migrations — the 5432
    `DIRECT_DATABASE_URL` exists but isn't wired into the schema, so correct migration
    runs depend on out-of-band config.
11. **`account_id` index coverage is uneven (low, perf).** `student_schedule_items`
    and `custom_events` index `account_id` (`schema.prisma:1327`,`:1360`), but most
    other tenant tables (students, courses, books, resources, etc.) have **no index on
    `account_id`** — every tenant-scoped list query does a filtered scan. At scale this
    is a performance risk given org filtering is the primary access pattern.

No truly dead models were found; all 56 models are referenced by other subsystems.
The `00000000000000_extensions_rls` baseline is real and load-bearing (not dead).
